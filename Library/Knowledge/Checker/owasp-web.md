# OWASP Web & API Security

**When to use:** Reviewing web applications with user input, HTTP endpoints, or API surfaces. Walk every review against these two lists.

---

## OWASP Web Top 10 (2021)

Map every code review against these 10 categories. Ask: "Which of these could this code be vulnerable to?"

| # | Risk | Key Attacks | Test For |
|---|------|-------------|----------|
| A01 | Broken Access Control | IDOR, missing function-level auth, CORS misconfiguration | Change object IDs, test unauthenticated access to privileged endpoints |
| A02 | Cryptographic Failures | Plaintext secrets, weak algorithms, poor key management | Check for hardcoded secrets, weak TLS, unencrypted PII |
| A03 | Injection | SQL, NoSQL, OS command, LDAP via untrusted input | Feed injection payloads through every input path |
| A04 | Insecure Design | Flaws in the design itself, missing threat modeling | Review architecture for missing security controls |
| A05 | Security Misconfiguration | Default configs, open storage, verbose errors, missing headers | Check response headers, error output, default credentials |
| A06 | Vulnerable Components | Known CVEs in dependencies | Run npm audit / pip-audit on every review |
| A07 | Auth Failures | Broken login, session management, credential stuffing | Test JWT none algorithm, session fixation, token reuse |
| A08 | Software Integrity | Insecure CI/CD, unsigned updates, deserialization | Check CI pipeline config, dependency integrity |
| A09 | Logging Failures | No audit trail, no alerting on suspicious activity | Verify security events are logged and alerted |
| A10 | SSRF | App fetches user-supplied URLs to internal services | Test internal URL access via user-controlled parameters |

**Reference:** https://owasp.org/Top10/ — https://cheatsheetseries.owasp.org/

## OWASP API Security Top 10 (2023)

APIs have unique attack patterns. Test every API endpoint against this list.

| # | Risk | Core Attack | Test |
|---|------|-------------|------|
| API1 | Broken Object Level Auth (BOLA) | Swap object IDs in every request | Change user/resource IDs, verify access denied |
| API2 | Broken Authentication | Token tampering, identity assumption | Send expired/tampered/missing tokens |
| API3 | Broken Property Level Auth | API returns too much, accepts too much | Request extra fields, try mass assignment |
| API4 | Unrestricted Resource Consumption | No rate limiting = DoS + cost explosion | Send 1,000 rapid requests, verify rate limiting |
| API5 | Broken Function Level Auth | Complex ACL hierarchies miss admin endpoints | Call admin endpoints as regular user |
| API6 | Unrestricted Business Flows | Automated abuse (ticket scalping, mass purchasing) | Automate a business flow, verify bot detection |
| API7 | SSRF | API fetches user-supplied URIs | Pass internal URLs as parameters |
| API8 | Security Misconfiguration | Missing headers, verbose errors, permissive CORS | Check response headers and error verbosity |
| API9 | Improper Inventory Management | Old API versions still live, debug endpoints exposed | Probe for /v1/, /debug/, /test/ endpoints |
| API10 | Unsafe API Consumption | Trusting third-party API data more than user input | Send malformed data pretending to be a trusted service |

**Reference:** https://owasp.org/API-Security/editions/2023/en/0x11-t10/

## Quick Attack Canaries

Feed these through every input path before calling a review complete:

```
SQLi:        ' OR '1'='1    |    ' UNION SELECT NULL --    |    1; DROP TABLE users --
XSS:         <script>alert('XSS')</script>    |    <img src=x onerror=alert('XSS')>
Cmd inject:  ; ls -la    |    | cat /etc/passwd    |    $(whoami)
SSRF:        http://169.254.169.254/latest/meta-data/    |    http://127.0.0.1
Path trav:   ../../../etc/passwd
```
