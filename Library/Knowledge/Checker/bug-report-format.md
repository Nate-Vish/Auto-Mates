# Bug Report Format

**When to use:** Any time Checker finds a vulnerability, bug, or failing test. Every finding must be documented well enough for Builder to reproduce and fix it without asking questions.

---

## The Attack Report Template

```markdown
### [BUG-ID] [Brief title]

**Severity:** Critical / High / Medium / Low
**Location:** file_path:line_number
**Found by:** [test name or manual attack]

**What happened:**
[1-2 sentences describing the failure]

**Reproduction steps:**
1. [Exact step]
2. [Exact step]
3. [Exact step]

**Expected behavior:**
[What should happen]

**Actual behavior:**
[What actually happens]

**Proof:**
[Test output, error message, screenshot, or curl command]

**Fix path:**
[Concrete suggestion for how to fix this]
```

---

## Severity Guide

| Severity | Definition | Example |
|----------|-----------|---------|
| **Critical** | Security vulnerability or data loss | SQL injection succeeds, passwords exposed, auth bypass |
| **High** | Feature completely broken | Login fails for all users, crash on valid input |
| **Medium** | Feature partially broken | Edge case fails, error message wrong |
| **Low** | Minor issue, workaround exists | Cosmetic, non-standard behavior in rare case |

---

## Writing Good Reproduction Steps

- **Be specific:** "Click the submit button" not "interact with the form"
- **Include exact input:** Show the actual payload/data used
- **State the starting condition:** "With a fresh session, no items in cart..."
- **Number every step:** Ordered list, no ambiguity
- **Include the failing test:** If you wrote one, reference it or paste the output

---

## Common Mistakes

| Mistake | Why It Fails | Better Approach |
|---------|-------------|-----------------|
| "It doesn't work" | No actionable information | Describe exactly what behavior is wrong |
| "Enter invalid data" | Missing specifics | State the exact payload used: `' OR '1'='1` |
| No expected vs actual | Reader can't tell what's wrong | Always include both |
| No fix suggestion | Checker's job includes fix direction | Always suggest a fix path |
| Testing the wrong thing | Bug is in test, not in code | Verify test independently before reporting |

---

## Checker Always Suggests a Fix

A report without a fix suggestion is incomplete. Even if Builder ultimately finds a better solution, Checker should always provide:
- What layer the fix belongs in (input validation, output encoding, auth layer, etc.)
- A concrete suggestion (parameterized queries, helmet middleware, denylist, etc.)
- Any documentation or cheatsheet references that apply
