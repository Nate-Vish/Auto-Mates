# Defense: Supply Chain & CI/CD Pipeline Security

**When to use:** When new dependencies are being added to a project, when reviewing GitHub Actions workflows, or when auditing CI/CD pipeline configuration.

---

## Supply Chain Security

### The Threat Landscape (2025-2026)

- 3,000+ malicious npm packages discovered in 2024
- 19.7% of AI-suggested packages are hallucinated (slopsquatting) — attackers register these hallucinated names
- Q1 2025: 56% of discovered malware targets data exfiltration
- September 2025: 18 popular packages (chalk, debug) compromised via phished maintainer credentials — 2.6B weekly downloads affected

### Attack Types

**Typosquatting:** Malicious packages with names similar to popular ones (`lodahs` not `lodash`). 71.2% of malicious packages use names with 10+ characters. 67.3% include dashes.

**Dependency Confusion:** Attacker publishes a public package with the same name as your internal private package, with a higher version number. Package managers resolve to the malicious public version.

**Slopsquatting (AI-driven):** AI code generators hallucinate package names. Attackers register those names with malicious code before anyone installs them legitimately.

**Compromised Maintainer:** Phishing or credential theft to take over a legitimate package's publishing account.

### Defensive Practices

**Immediate (do on every review):**
1. Pin exact versions — no `^` or `~` in production dependencies
2. Commit lockfiles to version control
3. Use `npm ci` (not `npm install`) in CI — installs from lockfile exactly
4. Run `npm audit` / `pip-audit` / equivalent — check for known CVEs

**For every new dependency added:**
- [ ] How old is this package? (< 6 months = higher risk)
- [ ] How many maintainers?
- [ ] Weekly download trends (sudden spike = suspicious)
- [ ] Does the name look like a typo of a popular package?
- [ ] Any hardcoded registry URLs bypassing the normal registry?

**Wait 7-14 days before adopting new packages** — most malicious packages are detected and pulled within days of publication.

### CVSS Severity Response

| CVSS | Severity | Action |
|------|----------|--------|
| 9.0-10.0 | Critical | Fix immediately, consider hotfix release |
| 7.0-8.9 | High | Fix this sprint (within 7 days) |
| 4.0-6.9 | Medium | Schedule fix (within 30 days) |
| 0.1-3.9 | Low | Track, fix in normal cycle |

## CI/CD Pipeline Security (GitHub Actions)

### Critical Risk 1: Script Injection

Untrusted input interpolated directly into `run:` blocks creates code execution:

```yaml
# VULNERABLE: PR title becomes a shell command
run: echo "Title: ${{ github.event.pull_request.title }}"

# SAFE: Use environment variable instead
env:
  TITLE: ${{ github.event.pull_request.title }}
run: echo "Title: $TITLE"
```

Never use `${{ github.event... }}` directly inside `run:` blocks.

### Critical Risk 2: Compromised Third-Party Actions

A single malicious action gets access to ALL repository secrets.

```yaml
# VULNERABLE: tag can be moved to point at malicious code
uses: some-action@v2

# SAFE: SHA cannot be changed after publish
uses: some-action@a1b2c3d4e5f6789abcdef...
```

**Always pin to full commit SHA.**

### Critical Risk 3: `pull_request_target` Trigger

`pull_request_target` runs with write access and secrets, but against the fork's code. A fork PR can access your production secrets.

**Rule:** Never use `pull_request_target` unless you fully understand the security implications. Use `pull_request` instead.

### Secret Handling

- Automatic redaction is NOT guaranteed for transformed secrets (Base64/URL-encoded)
- Structured data (JSON/YAML) containing secrets may not be fully masked
- Secrets in `run:` blocks may appear in logs

### Checker's CI/CD Review Checklist

When reviewing any GitHub Actions workflow:
- [ ] Search for `${{ github.event` — potential script injection points
- [ ] Check all `uses:` — are they SHA-pinned?
- [ ] Check `permissions:` — is it least-privilege?
- [ ] Search for secrets in `run:` blocks — any logged or transformed?
- [ ] Check for `pull_request_target` trigger — dangerous for forks
- [ ] Check that `GITHUB_TOKEN` permissions are scoped to minimum needed
- [ ] Verify no workflows auto-approve PRs
