# Defense: Security Headers, CORS, Cookies & Auth

**When to use:** Reviewing web server configuration, HTTP responses, authentication flows, JWT usage, OAuth implementations, or session management.

---

## HTTP Security Headers

### Must-Have Headers

| Header | Recommended Value | Purpose |
|--------|-------------------|---------|
| `Content-Security-Policy` | Restrict sources per resource type | Prevents XSS, injection, clickjacking |
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains; preload` | Forces HTTPS, prevents downgrade attacks |
| `X-Content-Type-Options` | `nosniff` | Prevents MIME type sniffing |
| `X-Frame-Options` | `DENY` | Prevents clickjacking (CSP frame-ancestors is modern alternative) |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Controls referrer leakage |
| `Permissions-Policy` | `geolocation=(), camera=(), microphone=()` | Restricts browser feature access |

### Headers to REMOVE

| Header | Why |
|--------|-----|
| `Server` | Reveals server software to attackers |
| `X-Powered-By` | Reveals tech stack |
| `X-AspNet-Version` | Reveals framework version |
| `X-XSS-Protection` | Set to `0` — the old filter can create vulnerabilities. Use CSP instead. |

### CSP Quick Start

```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'self'; frame-ancestors 'none';
```

### CORS Rules
- Never `Access-Control-Allow-Origin: *` on authenticated endpoints
- Whitelist specific origins: `Access-Control-Allow-Origin: https://yoursite.com`
- Validate `Origin` header server-side before reflecting it

### Cookie Security
Every auth cookie needs all three:
- `HttpOnly` — blocks XSS theft
- `Secure` — HTTPS only
- `SameSite=Strict` — prevents CSRF

### Implementation
- **Node.js/Express:** `app.use(helmet())`
- **Nginx:** `add_header "X-Frame-Options" "DENY" always;`
- **Apache:** `Header always set X-Frame-Options "DENY"` in .htaccess

### Checker's Header Test
1. Fetch the response headers
2. Check for all Must-Have headers
3. Check for headers that should be removed
4. Verify CSP is present and restrictive (not `unsafe-inline`, not `*`)
5. Verify cookies have HttpOnly, Secure, SameSite

**Testing tools:** https://observatory.mozilla.org | https://securityheaders.com

## Authentication: JWT Vulnerabilities

### 1. None Algorithm Attack
**Test:** Modify JWT header to `{"alg":"none"}`, remove signature, send request.
**If accepted:** Auth is completely bypassed.
**Fix:** Explicitly specify the expected algorithm during validation.

### 2. Weak Secrets
HMAC secrets < 64 chars can be cracked offline with Hashcat.
**Fix:** Minimum 64 characters, cryptographically random. Consider RSA over HMAC.

### 3. No Revocation
JWTs have no built-in logout. Token valid until expiry.
**Fix:** Implement a server-side token denylist (store SHA-256 digests of revoked tokens).

### 4. Token Storage
localStorage persists across sessions and is accessible to XSS.
**Fix:** Use HttpOnly cookies for JWT storage.

### 5. Information Disclosure
JWT payloads are Base64, not encrypted. Anyone can read claims.
**Fix:** Don't put sensitive data in JWT payload. Encrypt if necessary.

### JWT Test Checklist
- [ ] Can I use a JWT with algorithm "none"?
- [ ] Can I reuse a token after logout?
- [ ] Can I decode the JWT and see sensitive data?
- [ ] Can I brute-force the HMAC secret (< 64 chars)?

## OAuth Common Pitfalls

| Pitfall | Attack | Fix |
|---------|--------|-----|
| Missing `state` parameter | CSRF against OAuth flow | Always include and validate `state` |
| Open redirect in callback | Steal authorization code | Whitelist redirect URIs strictly |
| Token in URL fragment | Leaked via Referer header | Use code flow, not implicit flow |
| No PKCE for public clients | Authorization code interception | Require PKCE on all public clients |
| Insufficient scope | App requests more than needed | Request minimum required scopes |

## Session Management

| Issue | Risk | Fix |
|-------|------|-----|
| Session fixation | Attacker sets victim's session ID | Regenerate session ID after login |
| Long session expiry | Stolen sessions remain valid | Idle timeout + absolute timeout |
| No invalidation on logout | Logout doesn't actually log out | Server must destroy session on logout |
| Predictable session IDs | Session hijacking | Use cryptographically random generators |

### When NOT to Use JWT
If you don't need stateless architecture, traditional server-side sessions are simpler and more secure. They provide immediate revocation without a denylist and no client-side storage concerns.
