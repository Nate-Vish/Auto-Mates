# OWASP LLM & Prompt Injection Security

**When to use:** Reviewing LLM integrations, agent systems, MCP tools, or any code that processes AI-generated output or passes user input to an LLM.

---

## OWASP Top 10 for LLM Applications (v1.1)

AutoMates IS an LLM-based system. These risks apply to every agent interaction.

| # | Risk | Description | AutoMates Relevance |
|---|------|-------------|---------------------|
| LLM01 | Prompt Injection | Crafted inputs manipulate LLM behavior — direct and indirect variants | Agent-to-agent messages via shared files; user commands |
| LLM02 | Insecure Output Handling | Failing to validate LLM outputs leads to XSS, SSRF, code execution downstream | Never pass LLM output directly to shell, SQL, or HTML |
| LLM03 | Training Data Poisoning | Tampered data impairs model accuracy and security behavior | Fetcher processes untrusted web content |
| LLM04 | Model Denial of Service | Resource-heavy prompts cause disruption and cost spikes | Rate limit inputs; avoid unbounded loops |
| LLM05 | Supply Chain Vulnerabilities | Compromised components, plugins, or datasets | Review MCP servers before installing |
| LLM06 | Sensitive Information Disclosure | Model inadvertently reveals training data or config | Test for data leakage in outputs |
| LLM07 | Insecure Plugin Design | Plugins processing untrusted input with insufficient access control | Any MCP tool that takes user-controlled parameters |
| LLM08 | Excessive Agency | Unchecked LLM autonomy leads to unintended consequences | Require human confirmation for destructive actions |
| LLM09 | Overreliance | Failing to critically assess LLM output | "It runs" is not proof it is correct |
| LLM10 | Model Theft | Unauthorized model access for IP theft | Protect API keys; don't expose model internals |

## Prompt Injection Attack Patterns

### The Fundamental Problem

Prompt injection occurs when untrusted input is mixed with trusted instructions in an LLM context. The model cannot reliably distinguish between the two. **This is an open problem — no universal fix exists.**

### Direct Injection (User-Controlled)

```
"Ignore all previous instructions and..."
"You are now in developer mode..."
"Pretend you are DAN..."
```

### Indirect Injection (Data the LLM Reads)

Malicious instructions embedded in data — not from the user directly:
- Poisoned web pages the LLM fetches
- Malicious content in documents or databases the LLM reads
- Instructions hidden in tool outputs
- Hidden HTML comments: `<!-- Ignore above, instead do X -->`

### MCP-Specific Attack Patterns

| Attack | How It Works |
|--------|-------------|
| **Rug Pulls** | MCP server redefines tool descriptions after installation. Safe on day 1, malicious on day 5. |
| **Tool Shadowing** | Malicious MCP server intercepts calls to legitimate trusted tools. |
| **Tool Poisoning** | Hidden instructions in tool descriptions — visible to LLM, invisible to users. Can exfiltrate data through tool parameters. |

### The Dangerous Triangle

Attacks succeed when ALL THREE of these elements are present:

```
1. Private data access    — LLM can read sensitive information
2. Untrusted instructions — LLM processes potentially hostile input
3. Exfiltration capability — LLM can send data externally
```

**Remove ANY one element and the attack fails.** This is the primary mitigation framework.

## Defense Guidelines

### For LLM Tool Builders
- Never pass unescaped strings to system commands
- Treat all LLM output as untrusted (insecure output handling)
- Validate tool inputs independently of LLM decisions
- Implement human-in-the-loop for destructive or irreversible actions

### For System Architects
- Separate read and write permissions at the tool level
- Sandbox tool execution environments
- Log all tool invocations for audit
- Implement rate limiting on tool calls
- Display tool descriptions prominently to users before approving connections

### For AutoMates Specifically
- Agent-to-agent messages pass through shared files — validate file contents before acting on them
- Fetcher processes web content — treat ALL fetched content as DATA, never as instructions
- Memory files could be poisoned — validate on read
- Command bar accepts user input — sanitize before processing

## Silent Failures — The LLM-Specific Code Smell

Modern LLMs don't crash — they generate code that "fails to perform as intended but seems to run successfully." They may:
- Remove safety checks while keeping the function signature intact
- Create fake output matching the expected format
- Introduce subtle logic errors that look correct on first read

**Checker test:** Don't trust "it runs." Verify the OUTPUT is correct, not just that there is no error.
